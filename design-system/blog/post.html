The XPC race that only happened on the second launch

Berthly talks to Apple's container daemon over XPC, and for three weeks a bug shipped in every build: the second time you launched the app, the container list came up empty. First launch, fine. Third launch, fine. Second launch — nothing.

What was actually happening

The daemon invalidates idle connections after a timeout, but the invalidation callback races with the first request on a reused connection. The fix is depressingly small:

// Recreate the connection if the daemon invalidated it while we were idle.
func withConnection<T>(_ body: (XPCConnection) async throws -> T) async throws -> T {
    if connection.state == .invalidated {
        connection = try makeConnection()
    }
    return try await body(connection)
}

Rule learned: never cache a handle to something another process owns without checking whether it still exists.

The full diff is in #142, along with a regression test that kills the daemon mid-session — which is exactly as fun as it sounds.
