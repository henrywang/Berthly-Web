Berthly manages containers, and container imagery is crowded: whales, boxes, ships, cranes.
The name comes from *berth* — where a ship ties up — so the mark became a **dock bollard with a
mooring line made fast**. The app sits berthed at the dock.

The palette is brand blue `#2A6FDB` with rope and water tints `#8FB6F0` and `#B9D4FB` — the same
tokens this site uses.

## A parametric icon

The icon isn't drawn in a design tool; it's emitted by a Python script. One source
(`gen_icons.py`) owns the Apple-squircle math — a superellipse with `n = 5` — and generates every
variant: the full app icon, a simplified favicon with one heavy rope wrap for small sizes, an
`apple-touch-icon`, and a single-color glyph for the menu bar template.

Parametric beats hand-drawn here for one reason: when a size or a rule changes, you rerun the
script instead of re-exporting eight artboards.

## macOS 26 changed the rules

Berthly targets macOS 26 "Tahoe", and Tahoe changed how icons render. The classic Big Sur recipe —
an 824-point body centered on a 1024 canvas, transparent margin, baked drop shadow — now gets
scaled down and placed on a white system backdrop: the *icon jail* ring.

The fix: app-icon artwork is now generated **full-bleed** with no baked shadow, and the system
supplies the mask and shadow. The old margined masters survive only for the dmg volume icon, which
macOS still renders the old way.

## Rendering without dependencies

One more wrinkle: no `rsvg` or ImageMagick on the build machine, and macOS's two built-in SVG
rasterizers have *complementary* bugs — `qlmanage` honors SVG filters but mis-lays-out small
canvases, while NSImage/CoreSVG scales correctly but silently ignores filters. So the build
script routes each file through whichever rasterizer can actually render it.

The whole pipeline lives in
[`design/icon`](https://github.com/henrywang/Berthly/tree/main/design/icon) in the repo if you
want to borrow it.
